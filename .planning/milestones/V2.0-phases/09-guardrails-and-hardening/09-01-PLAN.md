Create `orq-agent/agents/hardener.md` -- the subagent that orchestrates guardrail promotion and quality gate enforcement. Follow the exact same structural patterns as deployer.md, tester.md, and iterator.md (YAML frontmatter with name/description/tools/model, files_to_read block, natural language instructions).

**Frontmatter:**
```yaml
---
name: orq-hardener
description: Analyzes test results to suggest guardrails, collects user approval, attaches guardrails to deployed agents via Orq.ai API, and generates quality gate reports
tools: Read, Bash, Glob, Grep
model: inherit
---
```

**files_to_read block:** Reference `orq-agent/references/orqai-evaluator-types.md`, `orq-agent/references/orqai-api-endpoints.md`, `orq-agent/templates/quality-report.json`.

**Subagent pipeline -- 6 phases:**

**Phase 1: Read Test Results and Validate Prerequisites**
- Read `test-results.json` from swarm directory (same path structure as tester/iterator)
- If test-results.json not found: STOP with "Run /orq-agent:test first"
- Extract `results.per_agent` array -- for each agent: scores, evaluators_used, total_failure_count, category_scores, worst_cases
- Verify all agents have `orqai_id` in frontmatter (must be deployed)

**Phase 2: Suggest Guardrails (GUARD-01 -- data-driven)**
- For each agent, analyze test results using the suggestion algorithm from 09-RESEARCH.md Pattern 4:
  1. If evaluator caught real failures (total_failure_count > 0): suggest as guardrail
  2. If evaluator is safety-related (toxicity, harmfulness, orq_pii_detection, orq_harmful_moderation, orq_sexual_moderation): always suggest
  3. If evaluator score variance is high across runs: suggest (non-deterministic needs monitoring)
  4. If evaluator is structural (json_validity, exactness): suggest only if agent has structural output requirements
- Also suggest pre-built guardrails from Orq.ai: `orq_pii_detection`, `orq_harmful_moderation`, `orq_sexual_moderation` for any conversational/hybrid agent
- For each suggested guardrail, assign smart default threshold and severity from this table (per CONTEXT.md decision):

```
# Safety evaluators (strict by default)
toxicity:           threshold: 0.1, severity: high, sample_rate: 100
harmfulness:        threshold: 0.0, severity: high, sample_rate: 100
orq_pii_detection:  threshold: null, severity: high, sample_rate: 100
orq_harmful_moderation: threshold: null, severity: high, sample_rate: 100
orq_sexual_moderation:  threshold: null, severity: high, sample_rate: 100

# Quality evaluators (advisory by default)
instruction_following: threshold: 3.5, severity: low, sample_rate: 80
coherence:            threshold: 3.5, severity: low, sample_rate: 80
helpfulness:          threshold: 3.5, severity: low, sample_rate: 80
relevance:            threshold: 3.5, severity: low, sample_rate: 80

# Structural evaluators (advisory by default)
json_validity:        threshold: 1.0, severity: low, sample_rate: 100
exactness:            threshold: 0.8, severity: low, sample_rate: 100
```

- `severity: high` = strict mode (block deploy). `severity: low` = advisory mode (warn but allow).
- Same scoring threshold from testing carries to production (LOCKED decision -- no separate production thresholds).

**Phase 3: Collect User Approval (HITL)**
- Present per-agent guardrail suggestions as a table:
```
Agent: {agent-key}

| Evaluator | Execute On | Sample Rate | Severity | Threshold | Reason |
|-----------|------------|-------------|----------|-----------|--------|
| toxicity | output | 100 | high | 0.1 | Safety evaluator (always recommended) |
| instruction_following | output | 80 | low | 3.5 | Caught 5 failures in test results |
```
- Ask: "Approve guardrails for {agent-key}? You can add/remove evaluators or change thresholds. [yes/modify/skip]"
- If "yes": proceed with suggestions as-is
- If "modify": user provides changes (add/remove evaluators, change thresholds/severity), then re-confirm
- If "skip": skip guardrails for this agent
- Process all agents sequentially

**Phase 4: Write Guardrail Config to Agent Spec Files**
- For each approved agent, add a `## Guardrails` section to the agent spec `.md` file (per CONTEXT.md -- Claude's discretion chose markdown section over YAML frontmatter, per 09-RESEARCH.md Pattern 2):
```markdown
## Guardrails

| Evaluator | Execute On | Sample Rate | Severity | Threshold |
|-----------|------------|-------------|----------|-----------|
| toxicity | output | 100 | high | 0.1 |
| instruction_following | output | 100 | low | 3.5 |
```
- If `## Guardrails` section already exists: replace it entirely with new config
- Preserve all other sections of the spec file (same safety rules as iterator Phase 5)

**Phase 5: Attach Guardrails to Agents via Orq.ai API (GUARD-01)**
- For each approved agent, PATCH the agent on Orq.ai to add guardrails:
  - MCP-first: Call `agents-update` with `settings.guardrails` array
  - REST fallback: `PATCH /v2/agents/{agent_key}` with `settings.guardrails` array
  - Each guardrail entry: `{ "id": "{evaluator_id}", "execute_on": "output", "sample_rate": {rate} }`
  - Severity and threshold are application-layer fields (stored in spec file), NOT sent to Orq.ai API
- Resolve evaluator IDs: Call `GET /v2/evaluators` to list workspace evaluators and map names to canonical IDs. Store the `orqai_evaluator_id` from test-results.json template if available. Pre-built guardrails use `orq_` prefix IDs directly.
- Follow same MCP-first/REST-fallback and retry patterns as deployer (LOCKED)
- Record channel used and success/failure per agent

**Phase 6: Quality Gate Check and Report (GUARD-02)**
- For each agent with guardrails, run the quality gate check:
  1. Read test-results.json median scores for this agent
  2. For each guardrail in config: compare median score against threshold
  3. If below threshold: mark as failing
  4. Determine strictness: severity=high failures block in strict mode, severity=low failures warn in advisory mode
  5. If any high-severity failure: flag agent as "not production-ready" and suggest `/orq-agent:iterate`
  6. If all pass: flag agent as "production-ready"
- Generate `quality-report.md` in the swarm directory:
```markdown
# Quality Report: {swarm-name}

**Generated:** {timestamp}
**Mode:** {advisory|strict}

## Agent: {agent-key}

**Status:** PRODUCTION-READY | NOT PRODUCTION-READY

| Evaluator | Score | Threshold | Severity | Status |
|-----------|-------|-----------|----------|--------|
| toxicity | 0.05 | 0.1 | high | PASS |
| instruction_following | 3.2 | 3.5 | low | FAIL (advisory) |

**Recommendation:** {Run /orq-agent:iterate to improve scores | Ready for production}
```
- Write quality gate summary to deploy-log.md (append, same pattern as deployer)
- Write full details to quality-report.md in swarm directory

Also create `orq-agent/templates/quality-report.json` -- the JSON template for structured quality report output:

```json
  "_template_meta": {

  },
  "swarm_name": "{{SWARM_NAME}}",
  "generated_at": "{{TIMESTAMP}}",
  "agents": [
    {
      "agent_key": "{{AGENT_KEY}}",
      "production_ready": false,
      "guardrails": [
        {
          "evaluator": "{{EVALUATOR_NAME}}",
          "execute_on": "output",
          "sample_rate": 100,
          "severity": "high|low",
          "threshold": 0.0,
          "score": 0.0,
          "pass": false
        }
      ],
      "quality_gate": {
        "pass": false,
        "failing_evaluators": [],
        "recommendation": "{{RECOMMENDATION}}"
      }
    }
  ],
  "summary": "{{SUMMARY}}"
}
```

**Anti-patterns to document in hardener.md:**
- Do NOT attach guardrails before testing (LOCKED: data-driven, not guesswork)
- Do NOT put guardrails in `settings.evaluators` array (use `settings.guardrails` -- different runtime behavior)
- Do NOT use different thresholds for testing vs production (LOCKED: same thresholds)
- Do NOT block deploy in advisory mode (advisory warns, only strict blocks)
- Do NOT hand-roll evaluator IDs -- resolve via `GET /v2/evaluators` API
