- name: orq-tester

**Role section:** Tester receives a swarm directory path and optional agent-key filter. Orchestrates the full test pipeline from dataset transformation through results reporting.

**The tester must implement an 8-phase pipeline:**

**Phase 1: Pre-check deployment**
- For each agent in the swarm (or single agent if filtered), read the agent spec file and check for `orqai_id` in YAML frontmatter (set by Phase 6 deployer)
- If any agent lacks `orqai_id`, report clear error: "Agent {name} not deployed. Run /orq-agent:deploy first." and STOP
- Collect agent list with orqai_id, spec content, and dataset file paths

**Phase 2: Parse V1.0 datasets**
- For each agent, locate `{agent-key}-dataset.md` (clean) and `{agent-key}-edge-dataset.md` (edge) in the `datasets/` directory
- Parse markdown tables: split rows by `|`, trim cells, skip header separator line (line with `---`)
- Extract eval pairs: ID, Input, Expected Output, Pass Criteria columns
- Tag clean dataset entries with category from their original category (happy-path, variation, boundary)
- Tag edge dataset entries with their category (adversarial, edge-case, stress)
- If agent has NO dataset files, log warning and skip that agent (Claude's discretion)

**Phase 3: Augment to minimum 30 examples**
- Count total examples per agent (clean + edge combined)
- If total >= 30, skip augmentation
- If total < 30, generate additional examples as variations:
  - Parameter swaps: change specific values while keeping input structure
  - Complexity variations: simpler and more detailed versions of existing inputs
  - Format variations: terse vs verbose phrasing of same request
  - Rephrasings: semantically equivalent but differently worded inputs
- Tag ALL augmented examples with `source: augmented` (LOCKED decision)
- Adapt expected outputs for augmented examples (not copy-paste from originals)
- Assign augmented examples to appropriate categories (variation or boundary)

**Phase 4: Merge and split datasets**
- Merge clean + edge + augmented examples into single dataset per agent
- Each example has: id, input, expected_output, pass_criteria, category (happy-path/variation/boundary/adversarial/edge-case/stress), source (original/augmented)
- Split 60/20/20 into train/test/holdout sets (LOCKED decision)
- Stratified split: maintain category distribution across splits
- Holdout set is reserved for Phase 8 — NOT used in Phase 7 experiments (LOCKED anti-pattern)

**Phase 5: Upload datasets to Orq.ai**
- Use `@orq-ai/node` SDK for dataset operations (REST-based, not MCP — per research Pitfall 4)
- For each agent, create dataset: `POST /v2/datasets` with name `test-{swarm}-{agent-key}`
- Upload rows to test split dataset: `POST /v2/datasets/{id}/rows`
- Each row in Orq.ai format:
  ```json
  {
    "inputs": {
      "text": "[input from eval pair]",
      "category": "[happy-path|variation|boundary|adversarial|edge-case|stress]",
      "source": "[original|augmented]",
      "eval_id": "[original eval pair ID]"
    },
    "messages": [
      { "role": "user", "content": "[input from eval pair]" }
    ],
    "expected_output": "[expected output from eval pair]"
  }
  ```
- Upload train split separately (needed for future use, not for Phase 7 experiments)
- Upload holdout split separately (reserved for Phase 8)
- Record platform dataset IDs for use in experiment execution (Phase 6 of pipeline)

**Phase 6: Infer agent roles and select evaluators**
- For each agent, classify role:
  - **Structural:** spec mentions JSON output, schema, structured data, extraction, formatting, parsing, or has json_schema tool type
  - **Conversational:** spec mentions conversation, chat, support, explanation, advice, or natural language response
  - **Hybrid:** has both structured output requirements and conversational elements
  - **Override:** `test_role: structural|conversational|hybrid` frontmatter in agent spec takes precedence (LOCKED decision)
- Select evaluators based on role (LOCKED decisions):
  - Structural: `json_validity` (threshold 1.0), `exactness` (0.8), `instruction_following` (0.8)
  - Conversational: `coherence` (0.7), `helpfulness` (0.7), `relevance` (0.7), `instruction_following` (0.8)
  - Hybrid: union of structural + conversational, deduplicated
- Apply category overlays (LOCKED decision):
  - adversarial/edge-case examples additionally get `toxicity` (threshold 0.1, lower=better) and `harmfulness` (threshold 0.0)
- Record selected evaluators per agent for experiment execution

**Phases 7-8 (Experiment execution and results) — documented as stubs:**
- Phase 7: Execute experiments 3x per agent (will be fully specified in Plan 02)
- Phase 8: Aggregate results and produce output (will be fully specified in Plan 02)

**MCP-first/REST-fallback pattern:** Inherit from deployer for agent retrieval during pre-check. Use REST-only for dataset operations (per research guidance). Evaluatorq handles its own API calls for experiments.

**Retry with exponential backoff:** Same pattern as deployer (1s base, 2^attempt, 500ms jitter, 30s cap, respect Retry-After on 429).

**Anti-patterns section (include these):**
- Do NOT run a single mega-experiment with all agents — run per-agent experiments
- Do NOT create platform evaluators for built-in types — reference by name
- Do NOT upload augmented examples without `source: augmented` tag
- Do NOT use holdout set during Phase 7 — reserved for Phase 8
- Do NOT block on individual agent test failure — continue testing others
- Do NOT average scores across different evaluator scales — report per-evaluator

**Output format:** The tester returns a structured result object per agent containing: agent_key, role, evaluators_used, dataset_id, example_counts (original, augmented, total, per_split), and placeholder for scores (filled by experiment execution).
