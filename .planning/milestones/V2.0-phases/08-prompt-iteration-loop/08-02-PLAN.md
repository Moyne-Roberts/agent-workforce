**Phase 6: Re-deploy changed agents to Orq.ai (ITER-04)**

After approved changes are written to local spec files (Phase 5), invoke the deployer subagent to update the changed agents on Orq.ai.

**Re-deploy process:**
1. The deployer subagent is invoked with the swarm directory path (same as the original deploy)
2. The deployer's existing idempotent create-or-update logic handles the rest:
   - It reads all agent spec files in the swarm
   - For each agent, it compares local spec against Orq.ai state
   - Only agents with changed `instructions` field will be updated (PATCHed)
   - Unchanged agents get status `unchanged` and are skipped
3. After deploy completes, the deployer updates `orqai_version` and `deployed_at` in the spec file's YAML frontmatter

**Key points:**
- Do NOT modify the deployer subagent -- it already handles selective updates naturally
- The deployer processes the full swarm manifest but only PATCHes what changed
- If the deployer reports any failed resources, log the failure but continue to re-test (the changed spec file is already written locally)
- Record which agents were re-deployed and their new version info

**Display during re-deploy:**
```
Re-deploying {N} changed agent(s)...
  {agent-key}: updated (version: {new_version})
  {agent-key}: updated (version: {new_version})
Re-deploy complete.
```

**Phase 7: Re-test changed agents on holdout split (ITER-05)**

After re-deployment, invoke the tester subagent to run experiments against changed agents using the holdout dataset split.

**Re-test process:**
1. Invoke the tester subagent with:
   - Swarm directory path
   - Agent-key filter: list of changed agent keys only (not the full swarm)
   - **Dataset split override: "holdout"** -- direct the tester to use holdout dataset IDs from `test-results.json` under `dataset.per_agent.{agent_key}.holdout_dataset_id`
2. The tester runs its standard pipeline but uses holdout dataset IDs instead of test dataset IDs:
   - Skip Phases 1-5 of tester (dataset already uploaded in Phase 7 original run)
   - Skip Phase 6 (evaluator selection already done)
   - Execute Phase 7 (experiments 3x) using holdout dataset
   - Execute Phase 8 (aggregate results)
3. Tester produces updated scores for changed agents only

**Holdout split parameter -- tester modification:**
The tester subagent (from Phase 7) currently uses the test split by default. For Phase 8 re-testing, the iterator must direct the tester to use the holdout split. This is achieved by:
- Passing the holdout dataset IDs directly to the tester when invoking it
- The holdout dataset IDs are stored in `test-results.json` at `dataset.per_agent.{agent_key}.holdout_dataset_id`
- The tester accepts these IDs and uses them for experiment execution instead of the test split IDs

**Before/after score comparison:**
After re-test completes, compute and display a comparison table per changed agent:

```markdown
### Re-Test Results: {agent-key}

| Evaluator | Before | After | Delta | Status |
|-----------|--------|-------|-------|--------|
| {name} | {old_median} | {new_median} | {delta}% | improved/unchanged/regressed |

**Bottleneck:** {old_bottleneck} -> {new_bottleneck} ({improvement}%)
```

**Flag regressions:** If any evaluator's median DECREASED after changes (even if still passing), flag it:
```
Warning: {evaluator} regressed from {old} to {new} on {agent-key}. Net bottleneck still improved.
```

This warns about collateral damage from prompt changes (research Pitfall 2: cascading changes breaking working sections).

**Re-test error handling:**
- If re-test fails for an agent: log the error, keep the old scores as "after" (treat as no improvement), continue
- If re-test times out: same treatment -- log and continue
- Re-test failures do not roll back the spec file changes (those are already written and deployed)
